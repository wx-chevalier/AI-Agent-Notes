## 一、CoT（Chain of Thought）思维链：让模型“先推理，后输出”

### 1. 核心原理

CoT 的本质是**强制模型显式输出推理步骤**，而不是直接给出结果。对于复杂任务（如多条件判断、嵌套 JSON 生成、逻辑推导），模型的“思考过程”能显著减少逻辑错误，提高输出准确率。

- **学术核心**：来自论文《Chain-of-Thought Prompting Elicits Reasoning in Large Language Models》，核心发现是：大模型在生成推理步骤后，对复杂任务的性能提升远超直接输出结果。
- **适用场景**：JSON 生成中涉及**多字段依赖、条件分支、数据映射**的场景（如从非结构化文本中提取多个关联字段生成 JSON）。

### 2. 落地关键：如何设计 CoT 提示词？

核心是**明确推理步骤**，并要求模型按步骤输出，最后再生成目标 JSON。结合 JSON 生成的通用模板：

```
任务：从用户提供的文本中提取文档信息，生成包含file_path、doc_type、content、retry_count的JSON。
要求：
1. 先分析用户输入的文本，识别其中的文档路径、文档类型、文档内容、重试次数；
2. 检查字段完整性：若某字段缺失，重试次数默认填0，文档类型默认填"text"；
3. 验证字段合法性：doc_type只能是["image", "text", "pdf"]，retry_count必须是非负整数；
4. 最后严格输出JSON，无额外内容。

用户输入："我有一个存储在/data/images/confidential.jpg的图片文件，里面包含机密内容，之前已经重试了2次。"
```

### 3. 模型输出示例（CoT 推理+JSON 结果）

```
# 思考过程
1. 分析用户输入：文档路径是"/data/images/confidential.jpg"，文档类型是"image"，内容是"机密内容"，重试次数是2；
2. 字段完整性检查：所有字段均存在，无需补充默认值；
3. 字段合法性检查：doc_type属于允许值，retry_count是正整数，符合要求；

# JSON输出
{"file_path": "/data/images/confidential.jpg", "doc_type": "image", "content": "机密内容", "retry_count": 2}
```

### 4. 核心优势与业务适配

- **优势**：
  - 解决复杂 JSON 的**字段依赖问题**（如 retry_count 是否超过 3 次需结合业务规则判断）；
  - 推理过程可**追溯**，便于调试模型输出错误的原因；
- **业务适配**：
  - 在**3D 模型 CNC 加工**场景中，可用于从工艺文档中提取刀具参数、加工路径、精度要求等字段生成 JSON；
  - 在**文档处理工作流**中，可用于从审核日志中提取任务 ID、审核结果、重试次数等字段生成 JSON。

## 二、Few-Shot Prompting（少样本提示）：用示例代替千言万语

### 1. 核心原理

Few-Shot 是**通过给模型展示少量（1~5 个）输入输出示例**，让模型快速理解任务要求，比纯自然语言描述更高效。核心逻辑是：**模型对示例的学习能力远强于对文字描述的理解能力**。

- **适用场景**：当任务具有**明确的输入输出映射关系**，但用自然语言难以清晰描述时（如特定格式的 JSON 生成、字段映射规则）。
- **关键原则**：示例需**覆盖核心场景**、**格式统一**、**无错误**。

### 2. 落地关键：如何设计 Few-Shot 示例？

结合 JSON 生成的通用模板，示例结构为：`用户输入 → 模型输出（JSON）`，通常放在提示词的开头。

```
任务：将用户输入的文档信息转换为包含file_path、doc_type、content、retry_count的JSON。
示例1：
用户输入："文本文件/data/texts/normal.txt，内容为'正常文本'，未重试过。"
输出JSON：{"file_path": "/data/texts/normal.txt", "doc_type": "text", "content": "正常文本", "retry_count": 0}

示例2：
用户输入："图片文件/data/images/fail.jpg，识别失败，已重试2次。"
输出JSON：{"file_path": "/data/images/fail.jpg", "doc_type": "image", "content": "识别失败", "retry_count": 2}

示例3：
用户输入："PDF文件/data/pdfs/confidential.pdf，内容为'机密资料'。"
输出JSON：{"file_path": "/data/pdfs/confidential.pdf", "doc_type": "pdf", "content": "机密资料", "retry_count": 0}

现在处理用户输入："我有一个图片文件/data/images/audit.jpg，内容为'待审核'，已经重试了1次。"
```

### 3. 核心优势与业务适配

- **优势**：
  - 解决**“自然语言描述不清任务规则”**的问题（如 doc_type 的映射规则、retry_count 的默认值）；
  - 大幅降低模型的**输出偏差**，比纯 CoT 更高效；
- **业务适配**：
  - 在**固件产品经理**场景中，可用于从需求文档中提取功能点、优先级、依赖模块等字段生成 JSON；
  - 在**AI Agent 开发**中，可用于定义工具调用的参数格式（如函数名、参数列表、返回值类型）。

### 4. 关键注意事项

- 示例数量：一般**3~5 个**足够，过多会增加提示词长度，降低模型效率；
- 示例覆盖度：需覆盖**正常场景、边界场景、异常场景**（如字段缺失、值不合法）；
- 格式统一：示例的输入输出格式必须与用户实际输入输出一致。

## 三、结构化输出工程（含 Self-Correction 自修正）：技术层强制约束 JSON

### 1. 核心原理

结构化输出工程是**通过技术手段强制模型输出符合要求的格式**，结合你的后端经验，核心工具是：

- **JSON Schema**：定义 JSON 的结构、字段类型、必填项、取值范围；
- **Pydantic**：Python 中用于数据验证的库，可基于 Schema 生成数据模型，自动校验模型输出；
- **Self-Correction**：当模型输出不符合 Schema 时，将**错误信息反馈给模型**，让模型自动修正后重新输出。
- **平台能力**：OpenAI 的 JSON Mode、Function Calling，Anthropic 的 Structured Outputs，从模型底层强制输出 JSON。

### 2. 落地关键步骤（结合 Python 后端经验）

#### 步骤 1：定义 JSON Schema 和 Pydantic 模型

这是**结构化输出的核心**，利用你熟悉的 JSON Schema 和 Pydantic，明确 JSON 的约束条件。

```python
from pydantic import BaseModel, Field, ValidationError
from typing import Literal

# 定义Pydantic模型（对应JSON Schema）
class DocumentInfo(BaseModel):
    file_path: str = Field(description="文档存储路径")
    doc_type: Literal["image", "text", "pdf"] = Field(description="文档类型")
    content: str = Field(description="文档内容")
    retry_count: int = Field(ge=0, le=3, description="重试次数，0~3次")

# 对应的JSON Schema（可用于OpenAI Function Calling）
document_schema = {
    "type": "object",
    "properties": {
        "file_path": {"type": "string", "description": "文档存储路径"},
        "doc_type": {"type": "string", "enum": ["image", "text", "pdf"], "description": "文档类型"},
        "content": {"type": "string", "description": "文档内容"},
        "retry_count": {"type": "integer", "minimum": 0, "maximum": 3, "description": "重试次数，0~3次"}
    },
    "required": ["file_path", "doc_type", "content", "retry_count"]
}
```

#### 步骤 2：使用平台结构化输出功能（如 OpenAI JSON Mode）

利用模型平台的内置功能，从底层强制输出 JSON，无需依赖提示词。

```python
from openai import OpenAI
client = OpenAI()

def get_structured_output(user_query: str) -> str:
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": "你是一个文档信息提取助手，严格按要求输出JSON。"},
            {"role": "user", "content": user_query}
        ],
        response_format={"type": "json_object"}  # 强制JSON输出
    )
    return response.choices[0].message.content
```

#### 步骤 3：Pydantic 校验与 Self-Correction 自修正

当模型输出的 JSON 通过语法校验，但**不符合内容约束**（如 retry_count=4，doc_type="docx"）时，使用 Pydantic 校验，并将错误信息反馈给模型，让模型自动修正。

```python
def validate_and_correct(user_query: str, max_retries: int = 3) -> DocumentInfo:
    for _ in range(max_retries):
        # 1. 获取模型输出的JSON字符串
        json_str = get_structured_output(user_query)
        try:
            # 2. 解析并校验JSON（Pydantic强制校验）
            doc_info = DocumentInfo.model_validate_json(json_str)
            return doc_info
        except ValidationError as e:
            # 3. 自修正：将错误信息反馈给模型，让模型重新输出
            error_msg = f"你的输出不符合要求，错误信息：{e.errors()}"
            user_query = f"{user_query}\n{error_msg}\n请重新输出符合要求的JSON。"
    raise ValueError(f"超过最大重试次数{max_retries}，无法生成符合要求的JSON")

# 测试
if __name__ == "__main__":
    user_query = "图片文件/data/images/fail.jpg，识别失败，已重试4次。"
    try:
        result = validate_and_correct(user_query)
        print(result.model_dump_json(indent=2))
    except ValueError as e:
        print(e)
```

#### 步骤 4：Self-Correction 的提示词设计

当校验失败时，反馈给模型的提示词需要包含**原始查询**和**错误信息**，让模型明确修正方向。

```
原始查询：图片文件/data/images/fail.jpg，识别失败，已重试4次。
错误信息：[{'type': 'less_than_equal', 'loc': ('retry_count',), 'msg': 'ensure this value is less than or equal to 3', 'input_value': 4, 'ctx': {'le': 3}}]
要求：请根据错误信息修正，重新输出符合要求的JSON，无额外内容。
```

### 3. 核心优势与业务适配

- **优势**：
  - **技术层保障**：比纯提示词更可靠，解决模型“明知故犯”的问题；
  - **自修正闭环**：无需人工介入，自动完成“生成 → 校验 → 修正”；
  - **与后端无缝集成**：Pydantic 模型可直接用于接口参数校验、数据库存储；
- **业务适配**：
  - 在**文档处理工作流**中，可用于校验工作流节点的输入输出 JSON；
  - 在**3D 模型 CNC 加工**中，可用于校验刀具参数、加工路径的 JSON 格式；
  - 在**AI Agent 开发**中，可用于校验工具调用的参数格式，确保 Agent 正确调用工具。

### 4. 关键注意事项

- **Schema 设计**：需**尽可能详细**，包含字段类型、必填项、取值范围、描述，减少模型歧义；
- **重试次数**：设置合理的最大重试次数（如 3 次），避免无限循环；
- **错误信息反馈**：需将 Pydantic 的错误信息**清晰、简洁**地反馈给模型，避免信息过载。

## 四、三者结合：终极 JSON 输出稳定方案

将 **CoT 思维链、Few-Shot 少样本提示、结构化输出工程** 三者结合，可实现**生产级别的 JSON 输出稳定性**，核心流程如下：

### 1. 提示词设计：CoT + Few-Shot

```
任务：从用户输入中提取文档信息，生成符合以下要求的JSON：
1. 字段：file_path（字符串）、doc_type（只能是image/text/pdf）、content（字符串）、retry_count（0~3的整数）；
2. 先输出思考过程，再输出JSON，无额外内容。

示例1：
用户输入："文本文件/data/texts/normal.txt，内容为'正常文本'，未重试过。"
思考过程：
1. 提取字段：file_path="/data/texts/normal.txt"，doc_type="text"，content="正常文本"，retry_count=0；
2. 校验：所有字段符合要求。
输出JSON：{"file_path": "/data/texts/normal.txt", "doc_type": "text", "content": "正常文本", "retry_count": 0}

示例2：
用户输入："图片文件/data/images/fail.jpg，识别失败，已重试4次。"
思考过程：
1. 提取字段：file_path="/data/images/fail.jpg"，doc_type="image"，content="识别失败"，retry_count=4；
2. 校验：retry_count=4超过最大值3，需要修正为3。
输出JSON：{"file_path": "/data/images/fail.jpg", "doc_type": "image", "content": "识别失败", "retry_count": 3}

现在处理用户输入："PDF文件/data/pdfs/confidential.pdf，内容为'机密资料'，已重试2次。"
```

### 2. 技术层约束：平台 JSON Mode + Pydantic 校验

- 使用 OpenAI JSON Mode 强制输出 JSON；
- 使用 Pydantic 模型校验内容合法性；
- 校验失败时，触发 Self-Correction 自修正。

### 3. 核心逻辑

1. **Few-Shot**：让模型快速理解任务规则和字段约束；
2. **CoT**：让模型显式推理，减少逻辑错误（如 retry_count 的修正）；
3. **结构化输出**：技术层强制 JSON 格式，Pydantic 校验内容，自修正解决剩余错误。

## 五、总结与落地建议

### 1. 技术对比

| 技术            | 核心作用                   | 适用场景                         | 可靠性 |
| --------------- | -------------------------- | -------------------------------- | ------ |
| CoT 思维链      | 减少复杂任务的逻辑错误     | 多字段依赖、条件分支的 JSON 生成 | 中     |
| Few-Shot 少样本 | 快速传递任务规则，降低偏差 | 输入输出映射明确的 JSON 生成     | 中高   |
| 结构化输出工程  | 技术层强制格式和内容约束   | 生产级 JSON 输出，要求 100%合规  | 高     |

### 2. 落地优先级

1. **快速落地**：使用 **Few-Shot + 平台 JSON Mode**，无需复杂代码，快速提升 JSON 稳定性；
2. **复杂任务**：增加 **CoT 思维链**，解决多字段依赖和逻辑推导问题；
3. **生产级保障**：增加 **Pydantic 校验 + Self-Correction**，实现闭环自修正。

### 3. 业务适配建议

- **固件产品经理**：结合 CoT 和结构化输出，从需求文档中提取功能点、优先级、依赖模块生成 JSON；
- **文档处理工作流**：结合 Few-Shot 和自修正，实现工作流节点的输入输出 JSON 校验；
