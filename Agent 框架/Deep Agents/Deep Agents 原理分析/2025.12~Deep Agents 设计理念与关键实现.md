# 2025.12~Deep Agents 设计理念与关键实现

## 一、设计理念：突破长周期任务处理瓶颈

在 LLM 驱动的智能代理技术发展中，传统"LLM 调用工具"的简单架构存在显著局限——面对长周期、多步骤的复杂任务时，往往因缺乏规划能力、上下文管理不足和任务拆分能力薄弱而表现"浅层化"。Deep Agents 的设计初衷正是解决这一核心问题，其理念源于对"Deep Research"、"Claude Code"等前沿应用的抽象与泛化，最终形成四大核心设计支柱：

1. **结构化规划能力**：通过任务拆解与进度追踪，将复杂目标转化为可执行的离散步骤；
2. **子代理协同**：通过专业化子代理隔离上下文，实现"主代理统筹+子代理深钻"的分层协作；
3. **文件系统赋能**：将大上下文 offload 至文件系统，突破 LLM 上下文窗口限制；
4. **精细化提示工程**：通过系统化提示词设计，规范代理行为与工具使用逻辑。

这些理念共同支撑了 Deep Agents 处理长周期任务的核心能力——从简单的工具调用者升级为具备规划、协作、记忆与迭代能力的"深度代理"。

## 二、核心架构与数据流设计

Deep Agents 基于 LangGraph 的`StateGraph`构建，以"状态中心化"为核心设计原则，实现任务全生命周期的连贯流转与可追溯性。

### 1. 状态对象：数据交互的核心载体

所有操作围绕**状态（State）对象**展开，其核心字段定义了代理的完整上下文：

- `messages`：对话消息列表（含用户输入、AI 回复、工具调用记录），类型为`list[BaseMessage]`；
- `todos`：结构化任务清单（由`TodoListMiddleware`维护），包含任务描述、状态与优先级；
- `files`：文件系统状态（由`FilesystemMiddleware`管理），记录文件路径、内容及元数据（创建时间、修改时间等）；
- `subagent_tasks`：子代理任务记录，存储子代理类型、任务描述、返回结果；
- `structured_response`：指定输出格式时生成的结构化数据（如 Pydantic 模型实例）。

状态对象的动态更新贯穿任务处理全流程，确保各组件（中间件、工具、子代理）间数据一致性。

### 2. 数据流全链路

当通过`agent.invoke(inputs)`触发任务时，数据流按以下步骤流转：

1. **输入初始化**  
   解析用户输入（通常为含`messages`的字典），合并至状态的`messages`字段，并注入系统提示词（用户自定义提示+基础提示的组合）。例如，研究类任务会自动附加"规划 → 执行 → 合成"的流程提示。

2. **中间件预处理**  
   默认中间件链依次作用于状态，完成初始化与能力注入：

   - `TodoListMiddleware`：初始化`todos`字段，添加任务规划工具；
   - `FilesystemMiddleware`：绑定文件存储后端（如`StateBackend`或`CompositeBackend`），注册文件操作工具；
   - `SubAgentMiddleware`：加载子代理列表，注册`task`工具用于子代理调用；
   - `SummarizationMiddleware`：监测上下文长度，当超过阈值（如 170k tokens）时自动压缩历史消息。

3. **工具调用与决策循环**  
   LLM 基于当前状态判断下一步行动：

   - 若需调用工具（文件操作、子代理、搜索等），生成工具调用指令并更新状态；
   - 工具执行结果（如文件内容、子代理返回值）实时写入状态的`messages`或`files`字段；
   - 重复"LLM 推理 → 工具调用 → 结果更新"循环，直至`todos`任务完成或模型决定终止。

4. **结果输出**  
   任务终止时，状态中的`messages`与`structured_response`被整理为最终输出，支持流式返回或一次性结果。

### 3. 典型场景数据流示例：子代理文件分析

以"主代理委派子代理分析文件"为例，数据流如下：

```
用户输入（"分析论文摘要"）→ 主代理状态初始化 → LLM决定调用子代理 →
SubAgentMiddleware生成子代理独立状态（复制基础配置+隔离上下文）→
子代理调用`read_file`工具读取目标文件（结果写入子代理状态）→
子代理处理后将分析结果写入文件（`write_file`工具）→
子代理终止，主代理通过`read_file`读取子代理输出文件 →
主代理整合结果至`messages`，生成最终回复
```

## 三、中间件架构：模块化能力扩展

Deep Agents 采用** middleware 架构**实现功能的模块化与可组合性，所有核心能力（规划、文件操作、子代理等）均通过中间件注入，支持按需扩展或裁剪。

### 1. 内置核心中间件

| 中间件名称                         | 核心功能                                           | 关键工具/字段                    |
| ---------------------------------- | -------------------------------------------------- | -------------------------------- |
| `TodoListMiddleware`               | 任务规划与进度追踪，支持动态调整任务清单           | `write_todos`工具、`todos`字段   |
| `FilesystemMiddleware`             | 提供文件系统操作能力，支持上下文 offload 至文件    | `ls`/`read_file`/`write_file`等  |
| `SubAgentMiddleware`               | 管理子代理生命周期，支持主-子代理通信与结果整合    | `task`工具、`subagent_tasks`字段 |
| `SummarizationMiddleware`          | 自动压缩超长上下文，避免超出 LLM token 限制        | 基于模型 token 阈值触发压缩      |
| `AnthropicPromptCachingMiddleware` | 缓存系统提示词（Anthropic 模型专用），降低调用成本 | 提示词哈希缓存                   |
| `HumanInTheLoopMiddleware`         | 支持人类介入审批关键步骤（如高危工具调用）         | 基于`interrupt_on`配置触发暂停   |

### 2. 中间件工作机制

- **加载逻辑**：通过`create_deep_agent(middleware=[...])`配置，默认包含`TodoListMiddleware`、`FilesystemMiddleware`等核心组件，用户可添加自定义中间件扩展功能。
- **工具注入**：中间件通过`tools`属性注册工具（如`FilesystemMiddleware`注册 7 个文件操作工具），工具调用结果自动更新至状态。
- **状态扩展**：中间件可扩展状态字段（如`FilesystemMiddleware`添加`files`字段），并通过`stream_channels`暴露给外部追踪。

### 3. 自定义中间件示例

通过继承`AgentMiddleware`，可快速扩展代理能力（如添加天气查询工具）：

```python
from langchain_core.tools import tool
from langchain.agents.middleware import AgentMiddleware

@tool
def get_weather(city: str) -> str:
    """获取城市天气"""
    return f"{city}天气：晴，25℃"

class WeatherMiddleware(AgentMiddleware):
    tools = [get_weather]  # 注册自定义工具

# 注入自定义中间件
agent = create_deep_agent(middleware=[WeatherMiddleware()])
```

## 四、核心能力实现

### 1. 规划与任务分解

通过`TodoListMiddleware`实现，核心逻辑包括：

- 提供`write_todos`工具，支持代理动态生成任务清单（如`["1. 搜索LangGraph定义", "2. 整理核心特性"]`）；
- 任务状态实时更新（未完成/已完成），支持根据新信息调整计划；
- 系统提示词中嵌入规划指引，确保复杂任务先分解再执行。

### 2. 文件系统管理

`FilesystemMiddleware`提供完整的文件操作能力，支持：

- 基础操作：`ls`（列出文件）、`read_file`（读取内容）、`write_file`（创建/覆盖文件）、`edit_file`（编辑内容）；
- 高级操作：`glob`（模式匹配查找文件）、`grep`（内容搜索）；
- 存储后端灵活切换：默认使用`StateBackend`（内存存储），可通过`CompositeBackend`配置混合存储（如`/memories/`路径持久化至`StoreBackend`）。

示例：持久化存储配置

```python
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend
from langgraph.store.memory import InMemoryStore

agent = create_deep_agent(
    backend=CompositeBackend(
        default=StateBackend(),  # 默认路径临时存储
        routes={"/memories/": StoreBackend(store=InMemoryStore())}  # /memories/路径持久化
    )
)
```

### 3. 子代理协同

`SubAgentMiddleware`支持主代理委派任务给专业化子代理，实现上下文隔离与并行处理：

- 子代理定义：通过字典或`CompiledSubAgent`类配置，包含`name`、`description`、`system_prompt`、`tools`等；
- 调用方式：主代理通过`task(subagent_type="weather_agent", input="东京天气")`工具触发；
- 通信机制：大输入/输出通过文件系统传递（如主代理写任务指令至`input.txt`，子代理写结果至`output.txt`），避免上下文膨胀。

示例：定义天气子代理

```python
subagents = [
    {
        "name": "weather_agent",
        "description": "获取城市天气",
        "system_prompt": "使用get_weather工具回答天气问题",
        "tools": [get_weather],
        "model": "gpt-4"
    }
]
agent = create_deep_agent(subagents=subagents)
```

### 4. 长期记忆

通过混合存储后端与`agent.md`文件实现跨会话记忆：

- 结构化记忆：`/memories/`路径下的文件通过`StoreBackend`持久化，保存用户偏好、项目知识等；
- 非结构化记忆：`agent.md`（全局`~/.deepagents/agent.md`与项目级`.deepagents/agent.md`）加载个性化提示，如编码风格、项目规范等。

## 五、评估与优化

Deep Agents 结合 Harbor 与 LangSmith 构建评估闭环，持续优化任务表现：

1. **数据收集**：LangSmith 记录所有 LLM 调用、工具调用与性能指标，Harbor 提供奖励分数区分成功/失败案例；
2. **模式识别**：分析失败案例（如工具误用、规划不足），通过`Trial Summary`统计完成率、工具使用频率等；
3. **针对性优化**：调整提示词（强化规划要求）、改进工具描述（添加示例）、选择更适合复杂任务的模型（如 Claude Sonnet）。

## 六、总结

Deep Agents 通过"状态中心化流转+中间件模块化扩展"的架构，实现了长周期复杂任务的结构化处理。其核心价值在于：

- **可扩展性**：通过中间件与子代理机制，支持按需添加工具与领域能力；
- **鲁棒性**：文件系统与摘要中间件解决上下文溢出问题，长期记忆确保跨会话连续性；
- **可观测性**：结合 LangSmith 与 Harbor，实现全流程追踪与数据驱动优化。

作为下一代智能代理框架，Deep Agents 为研究分析、代码开发、项目管理等复杂场景提供了标准化的解决方案，推动智能代理从"单次交互"向"持续任务处理"演进。
